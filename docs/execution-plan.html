ResearchMate Execution Plan

For agentic workers: REQUIRED SUB-SKILL: use a task-by-task workflow. 每个 agent 只接一个模块，先读 docs/handoff/index.html 和 docs/handoff/context-state.json，再开始改代码。

Goal: 在契约先行的基础上，把 ResearchMate 分模块实现成可上线、可演示、可评测的 AI 研究学习工作台。 Architecture: Next.js 负责用户界面，FastAPI 负责权限与 API，Python worker 负责长任务，Supabase/R2/Qdrant/Redis 存储各自边界清晰。 Tech Stack: Next.js、FastAPI、Supabase Postgres/Auth、Cloudflare R2、Qdrant、Redis、LangGraph、Docling、MarkItDown。

后续 agent 不应把 501 not implemented 当作错误。它代表当前阶段只定义契约。

1. 协作拆分原则

拆分原则与 docs/handoff/index.html 保持一致：先看上下文边界，再看人数。如果任务共享同一段阻塞上下文，就由同一个 agent 顺序完成，不为了并行而强拆。

同一 agent 应连续完成会共同修改同一批 API、表结构、状态机、索引管线或运行时上下文的任务；只有输入输出契约稳定、互不抢同一上下文时，才拆给不同 agent 并行。

2. 执行组计划

执行组负责范围执行方式依赖条件验收标准
Agent A：数据闭环主线Auth、Project、Storage、Upload、Parser、Index、Local Ask顺序完成，不拆分。该链路共享用户隔离、文档状态机、R2 对象、chunk 元数据、Qdrant payload 和本地 RAG 运行上下文。当前 API、DB、OpenAPI、TypeScript 契约稳定。用户 A 不能读取用户 B 数据；25MB 与 MIME 校验生效；文档可完成上传、解析、切分、入库和本地问答；Local Ask 的 citations 必须来自本地检索 chunk。
Agent B：来源策略与 Web 扩展Source Policy、Hybrid、Web Search、Crawl、Web Evidence顺序完成。Mode resolver、tool policy validator、web cache 与 citation 规则共享同一策略上下文，不能拆成多个 agent 抢改。Agent A 已交付 Local Ask 闭环。Local mode 阻止 web 工具；Hybrid 遵循 local-first；失败网页不得进入 citations；web evidence 有来源、时间和缓存状态。
Agent C：Quiz 输出链路Quiz generation、Quiz save、Quiz display contract可在 Agent A 完成 Parser + Index 后独立进行；不依赖 Web 扩展。文档 chunk、引用字段和项目归属校验稳定。选择题正好 4 个选项；答案、解释、引用来源都通过 schema 校验；普通用户只能看到自己的 quiz set。
Agent D：前端 MVPAsk、Library、Quiz 页面与前端状态管理可先基于契约开发壳层；最终联调必须等 Agent A 的认证、上传和本地问答接口稳定后顺序完成。API contract、shared TypeScript contracts、核心接口返回形状稳定。Ask、Library、Quiz 可完成主流程；普通用户界面没有 Trace 入口；错误态与加载态不泄露内部 trace 或 secret。
Agent E：Trace 与 EvalDeveloper Trace、RAG Eval、admin-only 观测入口顺序完成。Trace schema、run_id、tool_calls、citations 和 eval 指标共享同一观测上下文。Agent A/B/C 的 run、sources、quiz 输出稳定。Trace 仅管理员可见；所有 prompt、token、provider secret、用户隐私字段已脱敏；eval 指标可追溯到 run 与 citation。
Agent F：安全与生命周期收口删除、过期、限流、secret scan、发布前安全清单最后收口，不提前并行改核心链路。该组横跨 DB、R2、Qdrant、Redis、API 和前端权限。Agent A-E 的主要数据路径已经稳定。删除 project 会清理 DB、R2、Qdrant、Redis；限流与 quota guard 生效；公开仓库不包含真实密钥、本地数据或缓存产物。

Agent A：数据闭环主线

边界： 认证、项目、上传、解析、索引和本地问答共享用户隔离、文档状态机和检索上下文，因此由同一个 agent 顺序完成。

Agent B：来源策略与 Web 扩展

边界： 只在 Local Ask 闭环稳定后接入 source policy、hybrid、web search/crawl 和 web evidence。

Agent C：Quiz 输出链路

边界： 依赖 Parser + Index 的 chunk 与 citation 字段，但不依赖 Web 扩展。

Agent D：前端 MVP

边界： 可先做契约壳层，真实联调等待 Agent A 的核心接口稳定后顺序完成。

Agent E：Trace 与 Eval

边界： Trace schema、run、tool_calls、citations 和 eval 指标共享观测上下文，应顺序完成。

Agent F：安全与生命周期收口

边界： 横跨全部数据路径，必须在主要业务链路稳定后最后收口。

3. 调度策略

  • 禁止强拆：同一个状态机、同一组表、同一条数据闭环、同一套 prompt 或 trace schema 被多个任务共同依赖时，默认给同一个 agent 顺序完成。
  • 允许并行：Agent A 完成 Parser + Index 后，Agent C 可启动 Quiz；Agent A 完成本地问答接口后，Agent D 可进入真实联调；Agent B 可在 Local Ask 稳定后接入 Web 扩展。
  • 交接要求：每个 agent 完成后必须更新相关 HTML 文档、记录测试命令、说明未实现逻辑和下一位 agent 的输入契约。

4. 停止条件

任何 agent 遇到以下情况必须停下来问：

  • 需要修改 API 字段名称。
  • 需要修改数据库主键或 owner 字段。
  • 需要引入真实 secret。
  • 需要删除用户数据或迁移不可逆。
  • 需要让 Auto 默认联网。
  • 需要把 Trace 暴露给普通用户。
  • 需要把共享阻塞上下文拆给多个 agent。

5. 每个 agent 的交付格式

每个 agent 完成后必须更新：

  1. 相关代码和测试。
  2. docs/handoff/index.html。
  3. docs/handoff/context-state.json。
  4. docs/execution-plan.html 中本模块勾选项。

每次结束前运行：

pytest tests/test_project_scaffold.py -q
python skill/agent-context-html/scripts/validate_context_dashboard.py docs/handoff

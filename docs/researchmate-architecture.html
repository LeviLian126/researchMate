ResearchMate 架构与技术栈选择文档

版本：v1.0 状态：MVP 架构定稿 目标周期：8-10 周 文档类型：架构设计与技术栈选择

---

1. 架构目标

ResearchMate 不是本地 Demo，而是一个可正式上线的轻量 AI SaaS 项目。架构目标如下：

  1. 可上线：前端部署到 Vercel 或 Cloudflare Pages，后端和 worker 可部署到独立云平台。
  2. 安全：前端不泄露任何 LLM/Search/Embedding/Qdrant/R2 secret。
  3. 可溯源：每次回答和 quiz 都能追溯到 local page/slide/chunk 或 web URL。
  4. 成本可控：默认不联网，Hybrid local-first，Search/Crawl 有限流和缓存。
  5. 工程可展示：支持 Developer Trace，能展示 routing、retrieval、tool calls、validation。
  6. RAG 可评测：支持 retrieval hit、citation coverage、quiz grounding 等评测。
  7. 可扩展：后续可加 /report、错题本、flashcards、独立 MCP server。

4. 部署策略

4.1 前端

  • Next.js。
  • shadcn/ui。
  • 部署到 Vercel 或 Cloudflare Pages。
  • 前端只持有公开配置：Supabase anon key、public URL 等。
  • 前端绝不持有 LLM/Search/Embedding/Qdrant/R2 secret。

4.2 后端

后端先本地开发，部署平台先不绑定。正式部署候选：

  • Render
  • Railway
  • Fly.io
  • Google Cloud Run

文档不绑定平台，要求平台支持：

  • FastAPI 常驻服务。
  • worker service。
  • 环境变量。
  • 出站 HTTP 请求。
  • 可连接 Supabase、R2、Qdrant、Redis。

4.3 Worker

Worker 必须独立于前端函数：

  • 文档解析耗时长。
  • embedding 批处理可能超时。
  • crawler 需要重试。
  • RAG eval 是批处理任务。

因此不把解析和 embedding 放在 Vercel Functions 或 Cloudflare Workers 里。

5. Auth 与用户隔离

5.1 Auth 方案

使用 Supabase Auth：

  • Google OAuth。
  • GitHub OAuth。

5.2 用户隔离原则

所有业务表必须带 user_id 或能通过 project_id 关联到 user_id。

后端每次请求必须校验：

request user_id == resource owner user_id

5.3 Supabase RLS

如果前端直接访问 Supabase，需要启用 Row Level Security。MVP 更推荐：

  • 前端通过 Supabase Auth 登录。
  • 前端拿 session token。
  • 业务数据主要通过 FastAPI 后端访问。
  • 后端执行 user_id 权限校验。
  • Supabase service role key 只在后端环境变量。

5.4 OAuth 回调

需要配置：

  • Google OAuth callback。
  • GitHub OAuth callback。
  • Local dev callback。
  • Production callback。

6. 前端安全边界

6.1 Secret 管理

绝不暴露到前端：

  • NVIDIA / OpenAI-compatible LLM provider API key。
  • Serper/Tavily key。
  • Jina/Firecrawl key。
  • Qdrant API key。
  • R2 access key / secret key。
  • Supabase service role key。
  • Embedding provider key。

6.2 Next.js 环境变量规则

Next.js 中只有 NEXT_PUBLIC_ 前缀变量会被打进浏览器 bundle。任何 secret 都不能使用 NEXT_PUBLIC_ 前缀。

允许前端使用：

  • NEXT_PUBLIC_SUPABASE_URL
  • NEXT_PUBLIC_SUPABASE_ANON_KEY

禁止前端使用：

  • SUPABASE_SERVICE_ROLE_KEY
  • R2_SECRET_ACCESS_KEY
  • QDRANT_API_KEY
  • NVIDIA_API_KEY
  • NVIDIA_BASE_URL
  • NVIDIA_MODEL
  • SERPER_API_KEY

6.3 AI Provider 配置集中化

LLM API 相关配置统一从后端/worker 的 .env 读取，不在代码、Markdown 文档、前端环境变量或 Git 仓库中硬编码。

必需配置：

LLM_PROVIDER=nvidia
NVIDIA_API_KEY=nvapi-...
NVIDIA_BASE_URL=https://integrate.api.nvidia.com/v1
NVIDIA_MODEL=<selected_nvidia_model>

原则：

  • .env 保存真实 key 和真实 provider URL，并加入 .gitignore。
  • .env.example 只保存变量名和占位符，不保存真实 key。
  • 后端通过统一 LLM_PROVIDER、NVIDIA_BASE_URL、NVIDIA_MODEL 初始化 OpenAI-compatible client。
  • 切换模型或 provider 时，只改 .env，不改业务代码。
  • NVIDIA_BASE_URL 虽不是密钥，但仍只允许后端读取，避免前端绕过后端限流、日志、权限和安全校验直接调用模型。

6.4 Signed URL

上传流程：

  1. 前端请求后端创建 upload URL。
  2. 后端校验 user session。
  3. 后端生成 R2 signed upload URL。
  4. 前端直传 R2。
  5. 前端通知后端创建 document record。

下载/预览流程：

  1. 前端请求文件或 snippet。
  2. 后端校验 user_id。
  3. 后端返回短时效 signed read URL 或代理返回文本 snippet。

8. Source Mode 与 Task Type

8.1 SourceMode

type SourceMode = "auto" | "local_only" | "web_only" | "hybrid";

8.2 TaskType

type TaskType = "answer" | "quiz";

8.3 ExecutionPlan

type ExecutionPlan = {
  source_mode: SourceMode;
  task_type: TaskType;
  allowed_tools: string[];
  requires_local_docs: boolean;
  requires_web: boolean;
  output_schema: "GroundedAnswer" | "QuizSet";
};

8.4 设计原则

Mode 决定查哪里。
Task 决定做什么。

Quiz 不进入 Mode selector，因为 quiz 是 task，不是 source。

9. Tool Policy

9.1 Allowed tools by mode

local_only:
  - query_local_docs
  - generate_answer
  - generate_quiz
  - save_quiz

web_only:
  - search_web
  - read_url
  - crawl_url_optional
  - generate_answer

hybrid:
  - query_local_docs
  - search_web_if_needed
  - read_url_if_needed
  - evidence_fusion
  - generate_answer
  - generate_quiz_local_first

auto:
  - determined by SourcePolicyResolver

9.2 SourcePolicyValidator

检查：

  • local_only 不得引用 web。
  • web_only 不得使用 local evidence。
  • hybrid 必须区分 local 和 web evidence。
  • auto 必须记录 resolved_mode。
  • quiz 必须绑定 local source citation。

12. RAG 设计

12.1 Qdrant collection

Collection: researchmate_chunks

Payload fields：

{
  "user_id": "...",
  "project_id": "...",
  "document_id": "...",
  "source_type": "local_doc | web_page",
  "page_no": 12,
  "slide_no": null,
  "url": null,
  "title": "...",
  "created_at": "..."
}

12.2 Retrieval filters

所有检索必须带：

user_id == current_user_id
project_id == current_project_id

Local retrieval：

source_type == local_doc

Web retrieval：

source_type == web_page

12.3 Embedding provider abstraction

Embedding provider 不绑定死，并通过 .env 配置切换。优先便宜、稳定、易替换：

  • NVIDIA embedding endpoint，如 nv-embed-v1，通过 NVIDIA_BASE_URL 调用 /v1/embeddings。
  • Jina Embeddings。
  • Voyage AI。
  • 后续可接本地 embedding。

建议配置：

EMBEDDING_PROVIDER=nvidia
NVIDIA_EMBEDDING_MODEL=nvidia/nv-embed-v1

接口：

class EmbeddingProvider:
    def embed_texts(self, texts: list[str]) -> list[list[float]]:
        ...

13. Web Search 与 Crawl

13.1 Provider strategy

能力主方案fallback
SearchSerperTavily
URL ReaderJina ReaderFirecrawl
Complex crawlFirecrawlCrawl4AI optional

13.2 成本控制

  • 默认不联网。
  • Hybrid local-first。
  • Search result cache。
  • URL content cache。
  • Per-user daily search limit。
  • Provider quota guard。
  • 超额时提示用户切换 Local mode。

13.3 Web evidence 入库

Web 资料默认作为临时 evidence：

  • 存 text snapshot。
  • 存 URL。
  • 存 retrieved_at。
  • 存 title。
  • 可在 7 天后随项目清理。

16. 删除与数据生命周期

16.1 删除策略

支持混合策略：

  1. 用户手动删除项目。
  2. 默认 7 天自动过期删除。

16.2 删除范围

删除项目需要清理：

  • projects record
  • documents records
  • document_pages
  • chunks metadata
  • Qdrant points
  • R2 objects
  • messages
  • quiz_sets
  • quiz_questions
  • ask_runs
  • tool_calls
  • Redis cache/job status

16.3 删除流程

User deletes project / expiration scheduler triggers
  |
  v
create deletion_job
  |
  v
mark project as deleting
  |
  v
delete R2 objects
  |
  v
delete Qdrant vectors
  |
  v
delete DB records or soft delete
  |
  v
clear Redis cache
  |
  v
mark deletion_job completed

16.4 失败处理

  • 如果 R2 删除失败，job 保持 failed，可重试。
  • 如果 Qdrant 删除失败，保留 deletion job。
  • 不允许部分失败静默通过。
  • Developer trace 或 admin log 可看到失败原因。

Search 超额provider quota 用尽今日联网额度不足降级 Local
Crawl 失败URL 不可读部分网页读取失败跳过该 URL
LLM JSON 错输出不符合 schema生成失败，请重试repair retry
citation 缺失answer 无来源生成失败，请重试validator reject
local_only 越权联网tool policy 错误不应发生SourcePolicyValidator fail

18. 安全设计

18.1 API key 防护

  • 所有 provider key 只在后端/worker。
  • 前端不接触 secret。
  • 不把 secret 写入 logs。
  • 不把 secret 放入 NEXT_PUBLIC 环境变量。

18.2 CORS

  • 后端只允许指定前端域名。
  • 本地开发单独配置 localhost。

18.3 CSRF

如果使用 cookie/session，需要 CSRF 防护。若使用 bearer token，则重点防 token 泄露和 CORS。

18.4 Prompt injection

Web/local content 不得修改：

  • Source policy。
  • Tool policy。
  • User permissions。
  • System prompt。
  • API keys。

处理策略：

  • 把 retrieved content 明确标记为 untrusted source text。
  • Tool calls 只由 graph 节点触发。
  • LLM 不直接决定越权工具。

18.5 文件上传安全

  • 文件类型白名单。
  • MIME 检查。
  • 文件大小限制。
  • object key 随机化。
  • 不执行用户上传文件。
  • 可选病毒扫描作为 Phase 2。

## 1. 架构目标

ResearchMate 不是本地 Demo，而是一个可正式上线的轻量 AI SaaS 项目。架构目标如下：

1. **可上线**：前端部署到 Vercel 或 Cloudflare Pages，后端和 worker 可部署到独立云平台。
2. **安全**：前端不泄露任何 LLM/Search/Embedding/Qdrant/R2 secret。
3. **可溯源**：每次回答和 quiz 都能追溯到 local page/slide/chunk 或 web URL。
4. **成本可控**：默认不联网，Hybrid local-first，Search/Crawl 有限流和缓存。
5. **工程可展示**：支持 Developer Trace，能展示 routing、retrieval、tool calls、validation。
6. **RAG 可评测**：支持 retrieval hit、citation coverage、quiz grounding 等评测。
7. **可扩展**：后续可加 `/report`、错题本、flashcards、独立 MCP server。

## 4. 部署策略

### 4.1 前端

- Next.js。
- shadcn/ui。
- 部署到 Vercel 或 Cloudflare Pages。
- 前端只持有公开配置：Supabase anon key、public URL 等。
- 前端绝不持有 LLM/Search/Embedding/Qdrant/R2 secret。

### 4.2 后端

后端先本地开发，部署平台先不绑定。正式部署候选：

- Render
- Railway
- Fly.io
- Google Cloud Run

文档不绑定平台，要求平台支持：

- FastAPI 常驻服务。
- worker service。
- 环境变量。
- 出站 HTTP 请求。
- 可连接 Supabase、R2、Qdrant、Redis。

### 4.3 Worker

Worker 必须独立于前端函数：

- 文档解析耗时长。
- embedding 批处理可能超时。
- crawler 需要重试。
- RAG eval 是批处理任务。

因此不把解析和 embedding 放在 Vercel Functions 或 Cloudflare Workers 里。

## 5. Auth 与用户隔离

### 5.1 Auth 方案

使用 Supabase Auth：

- Google OAuth。
- GitHub OAuth。

### 5.2 用户隔离原则

所有业务表必须带 `user_id` 或能通过 `project_id` 关联到 `user_id`。

后端每次请求必须校验：

```text
request user_id == resource owner user_id
```

### 5.3 Supabase RLS

如果前端直接访问 Supabase，需要启用 Row Level Security。MVP 更推荐：

- 前端通过 Supabase Auth 登录。
- 前端拿 session token。
- 业务数据主要通过 FastAPI 后端访问。
- 后端执行 user_id 权限校验。
- Supabase service role key 只在后端环境变量。

### 5.4 OAuth 回调

需要配置：

- Google OAuth callback。
- GitHub OAuth callback。
- Local dev callback。
- Production callback。

## 6. 前端安全边界

### 6.1 Secret 管理

绝不暴露到前端：

- NVIDIA / OpenAI-compatible LLM provider API key。
- Serper/Tavily key。
- Jina/Firecrawl key。
- Qdrant API key。
- R2 access key / secret key。
- Supabase service role key。
- Embedding provider key。

### 6.2 Next.js 环境变量规则

Next.js 中只有 `NEXT_PUBLIC_` 前缀变量会被打进浏览器 bundle。任何 secret 都不能使用 `NEXT_PUBLIC_` 前缀。

允许前端使用：

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

禁止前端使用：

- `SUPABASE_SERVICE_ROLE_KEY`
- `R2_SECRET_ACCESS_KEY`
- `QDRANT_API_KEY`
- `NVIDIA_API_KEY`
- `NVIDIA_BASE_URL`
- `NVIDIA_MODEL`
- `SERPER_API_KEY`

### 6.3 AI Provider 配置集中化

LLM API 相关配置统一从后端/worker 的 `.env` 读取，不在代码、Markdown 文档、前端环境变量或 Git 仓库中硬编码。

必需配置：

```env
LLM_PROVIDER=nvidia
NVIDIA_API_KEY=nvapi-...
NVIDIA_BASE_URL=https://integrate.api.nvidia.com/v1
NVIDIA_MODEL=<selected_nvidia_model>
```

原则：

- `.env` 保存真实 key 和真实 provider URL，并加入 `.gitignore`。
- `.env.example` 只保存变量名和占位符，不保存真实 key。
- 后端通过统一 `LLM_PROVIDER`、`NVIDIA_BASE_URL`、`NVIDIA_MODEL` 初始化 OpenAI-compatible client。
- 切换模型或 provider 时，只改 `.env`，不改业务代码。
- `NVIDIA_BASE_URL` 虽不是密钥，但仍只允许后端读取，避免前端绕过后端限流、日志、权限和安全校验直接调用模型。

### 6.4 Signed URL

上传流程：

1. 前端请求后端创建 upload URL。
2. 后端校验 user session。
3. 后端生成 R2 signed upload URL。
4. 前端直传 R2。
5. 前端通知后端创建 document record。

下载/预览流程：

1. 前端请求文件或 snippet。
2. 后端校验 user_id。
3. 后端返回短时效 signed read URL 或代理返回文本 snippet。

## 9. Tool Policy

### 9.1 Allowed tools by mode

```text
local_only:
  - query_local_docs
  - generate_answer
  - generate_quiz
  - save_quiz

web_only:
  - search_web
  - read_url
  - crawl_url_optional
  - generate_answer

  - query_local_docs
  - search_web_if_needed
  - read_url_if_needed
  - evidence_fusion
  - generate_answer
  - generate_quiz_local_first

auto:
  - determined by SourcePolicyResolver
```

### 9.2 SourcePolicyValidator

检查：

- local_only 不得引用 web。
- web_only 不得使用 local evidence。
- hybrid 必须区分 local 和 web evidence。
- auto 必须记录 resolved_mode。
- quiz 必须绑定 local source citation。

## 12. RAG 设计

### 12.1 Qdrant collection

Collection: `researchmate_chunks`

Payload fields：

```json
{
  "user_id": "...",
  "project_id": "...",
  "document_id": "...",
  "source_type": "local_doc | web_page",
  "page_no": 12,
  "slide_no": null,
  "url": null,
  "title": "...",
  "created_at": "..."

```

### 12.2 Retrieval filters

所有检索必须带：

```text
user_id == current_user_id
project_id == current_project_id
```

Local retrieval：

```text
source_type == local_doc
```

Web retrieval：

```text
source_type == web_page
```

### 12.3 Embedding provider abstraction

Embedding provider 不绑定死，并通过 `.env` 配置切换。优先便宜、稳定、易替换：

- NVIDIA embedding endpoint，如 `nv-embed-v1`，通过 `NVIDIA_BASE_URL` 调用 `/v1/embeddings`。
- Jina Embeddings。
- Voyage AI。
- 后续可接本地 embedding。

建议配置：

```env
EMBEDDING_PROVIDER=nvidia
NVIDIA_EMBEDDING_MODEL=nvidia/nv-embed-v1
```

接口：

```python
class EmbeddingProvider:
    def embed_texts(self, texts: list[str]) -> list[list[float]]:
        ...
```

## 16. 删除与数据生命周期

### 16.1 删除策略

支持混合策略：

1. 用户手动删除项目。
2. 默认 7 天自动过期删除。

### 16.2 删除范围

删除项目需要清理：

- projects record
- documents records
- document_pages
- chunks metadata
- Qdrant points
- R2 objects
- messages
- quiz_sets
- quiz_questions
- ask_runs
- tool_calls
- Redis cache/job status

### 16.3 删除流程

```text
User deletes project / expiration scheduler triggers
  |
  v
create deletion_job
  |
  v
mark project as deleting
  |
  v
delete R2 objects
  |
  v
delete Qdrant vectors
  |
  v
delete DB records or soft delete
  |
  v
clear Redis cache
  |
  v
mark deletion_job completed
```

### 16.4 失败处理

- 如果 R2 删除失败，job 保持 failed，可重试。
- 如果 Qdrant 删除失败，保留 deletion job。
- 不允许部分失败静默通过。
- Developer trace 或 admin log 可看到失败原因。

## 17. 错误与救援路径

| 错误 | 触发 | 用户看到 | 系统处理 |
|---|---|---|---|
| OAuth 失败 | provider callback 错误 | 登录失败，请重试 | 记录 auth error |
| 上传失败 | signed URL 过期/网络失败 | 上传失败，可重试 | 清理 pending document |
| 文件过大 | 超过限制 | 文件超过大小限制 | 拒绝上传 |
| parser 失败 | PDF 复杂/损坏 | 解析失败，可换文件 | 标记 document failed |
| embedding 失败 | provider 超额/超时 | 文档索引失败 | retry / fallback |
| Qdrant upsert 失败 | 网络或 schema 错误 | 索引失败 | retry job |
| Search 超额 | provider quota 用尽 | 今日联网额度不足 | 降级 Local |
| Crawl 失败 | URL 不可读 | 部分网页读取失败 | 跳过该 URL |
| LLM JSON 错 | 输出不符合 schema | 生成失败，请重试 | repair retry |
| citation 缺失 | answer 无来源 | 生成失败，请重试 | validator reject |
| local_only 越权联网 | tool policy 错误 | 不应发生 | SourcePolicyValidator fail |

## 18. 安全设计

### 18.1 API key 防护

- 所有 provider key 只在后端/worker。
- 前端不接触 secret。
- 不把 secret 写入 logs。
- 不把 secret 放入 NEXT_PUBLIC 环境变量。

### 18.2 CORS

- 后端只允许指定前端域名。
- 本地开发单独配置 localhost。

### 18.3 CSRF

如果使用 cookie/session，需要 CSRF 防护。若使用 bearer token，则重点防 token 泄露和 CORS。

### 18.4 Prompt injection

Web/local content 不得修改：

- Source policy。
- Tool policy。
- User permissions。
- System prompt。
- API keys。

处理策略：

- 把 retrieved content 明确标记为 untrusted source text。
- Tool calls 只由 graph 节点触发。
- LLM 不直接决定越权工具。

### 18.5 文件上传安全

- 文件类型白名单。
- MIME 检查。
- 文件大小限制。
- object key 随机化。
- 不执行用户上传文件。
- 可选病毒扫描作为 Phase 2。

### 23.2 Environment variables

Frontend `.env.local`:

```env
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
NEXT_PUBLIC_API_BASE_URL=
```

Backend / Worker `.env`:

```env
# Supabase
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=

# Cloudflare R2
R2_ACCOUNT_ID=
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_BUCKET=

# Vector DB
QDRANT_URL=
QDRANT_API_KEY=

# LLM provider: NVIDIA NIM / OpenAI-compatible API
LLM_PROVIDER=nvidia
NVIDIA_API_KEY=nvapi-...
NVIDIA_BASE_URL=https://integrate.api.nvidia.com/v1
NVIDIA_MODEL=<selected_nvidia_model>

# Embedding provider
EMBEDDING_PROVIDER=nvidia
NVIDIA_EMBEDDING_MODEL=nvidia/nv-embed-v1

# Search / crawl providers
SERPER_API_KEY=
TAVILY_API_KEY=
JINA_API_KEY=
FIRECRAWL_API_KEY=

# Cache / rate limit / jobs
REDIS_URL=
```

`.env` 管理原则：

- 真实值只写入本地、staging、production 的 `.env` 或云平台 secret manager。
- 仓库只提交 `.env.example`，并使用 `nvapi-...`、`<selected_nvidia_model>` 等占位符。
- 后端和 worker 只通过配置层读取 URL、model、API key；业务代码不得硬编码 provider URL 或 key。
- 切换 NVIDIA 模型、Base URL、搜索 provider、embedding provider 时，只改 `.env`。

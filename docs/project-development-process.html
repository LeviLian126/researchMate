ResearchMate 项目开发流程规范

适用项目：ResearchMate
适用目标：个人开发但按大厂级流程推进，最终可正式上线、可写进简历、可面试展示
版本：v1.0
生成日期：2026-05-25

---

0. 总原则

ResearchMate 不是一次性 Demo，而是一个需要正式上线的 AI 应用项目。开发流程应遵循：

  1. 先契约，后实现：先定 API、数据模型、页面行为，再写前后端代码。
  2. 先纵向闭环，后横向扩展：先跑通“上传文件 → 解析 → 向量化 → 提问 → 引用回答”这一条链，再扩展 Web、Hybrid、Quiz。
  3. 先可靠，后酷炫：Agent、RAG、MCP、Trace 都要服务于用户任务，不为了显摆而堆技术。
  4. 默认安全：用户文件、API Key、OAuth、对象存储、向量库、Trace 全部按正式上线标准设计。
  5. 每阶段有验收标准：没有验收标准的任务不进入开发。
  6. 每周必须有可演示结果：避免长期只写底层代码，最后无法展示。
  7. 成本可控：默认 Local-first，Web-on-demand，所有外部 API 做限流、缓存和降级。

参考依据：

  • DevOps 生命周期通常包含 plan、code、build、test、release、deploy、operate、monitor 等阶段。
  • OpenAPI 用于在看源码前明确 HTTP API 的能力和语义。
  • GitHub Actions 可用于自动化 build/test/CI。
  • OWASP ASVS 可作为 Web 应用安全控制和测试要求的参考。
  • Twelve-Factor App 建议将配置存储在环境变量中，避免把环境配置写死在代码仓库。

4. 阶段 2：API Contract

目标

在写前端和后端前，冻结接口契约。前端、后端、worker、测试都围绕同一份 API Spec 开发。

主要产物

  • API_SPEC.md
  • openapi.yaml
  • 错误码表
  • API 权限表
  • 请求/响应 schema

必须定义的 API

模块API
AuthGET /me
ProjectPOST /projects, GET /projects, GET /projects/{id}, DELETE /projects/{id}
UploadPOST /documents/upload-url
DocumentPOST /documents, GET /projects/{id}/documents, GET /documents/{id}, DELETE /documents/{id}
JobGET /jobs/{id}
AskPOST /ask
QuizPOST /quiz
SourcesGET /runs/{id}/sources
TraceGET /dev/traces/{trace_id}
HealthGET /healthz

API 设计原则

  1. 所有业务 API 必须带认证。
  2. 所有 project/document/run 查询必须校验 user_id。
  3. 长任务只返回 job_id，不在请求里同步等完。
  4. Ask 响应必须包含 mode 和 sources。
  5. Quiz 响应必须包含题目、答案、解析、引用来源。
  6. 错误响应必须统一格式。
  7. API 版本从 /api/v1 开始。

统一错误响应

{
  "error": {
    "code": "DOCUMENT_NOT_INDEXED",
    "message": "Document is still being processed.",
    "request_id": "req_..."
  }
}

Tips

  • API Spec 不要只写 endpoint，要写状态码、错误码、权限和示例。
  • 文件上传不要让前端拿 R2 secret，要让后端生成 signed upload URL。
  • Ask 不要直接返回一堆模型文本，要返回结构化字段。
  • Developer Trace API 必须只允许开发者或管理员访问。
  • OpenAPI 生成后，可以用它生成前端类型或 API client。

验收标准

  • [ ] 所有 MVP API 都在 openapi.yaml 里定义。
  • [ ] 每个 API 有 request、response、error examples。
  • [ ] 所有需要认证的 API 都标记 auth requirement。
  • [ ] Ask / Quiz schema 可被前端直接渲染。
  • [ ] API 错误码覆盖上传失败、解析中、检索为空、外部 API 超额、权限失败。
  • [ ] 前后端可以基于 mock API 并行开发。

5. 阶段 3：数据模型与权限设计

目标

确定数据库表、索引、权限隔离、删除策略、Qdrant payload 映射。

主要产物

  • DB_SCHEMA.md
  • ERD
  • Supabase migration SQL
  • RLS policy 草案
  • Qdrant collection schema

核心表

表作用
profiles用户资料
projects项目
documents文件元数据
document_pagesPDF 页 / PPT slide / DOCX section
chunkschunk 元数据
messages对话消息
ask_runs一次 Ask 执行
tool_calls工具调用记录
citations引用关系
quiz_setsQuiz 批次
quiz_questionsQuiz 题目
jobs异步任务
deletion_jobs删除任务
api_usage用户 API 使用量 / 限流统计

关键字段

所有用户数据表至少要有：

user_id uuid not null
created_at timestamptz not null
updated_at timestamptz
deleted_at timestamptz

权限原则

  • 用户只能访问自己的 project。
  • document、chunk、quiz、trace 都通过 project 绑定 user。
  • 后端每次查询都带 user_id。
  • Supabase RLS 能开则开，但后端也要做强校验。
  • service role key 只在后端/worker。

Qdrant payload

{
  "user_id": "...",
  "project_id": "...",
  "document_id": "...",
  "chunk_id": "...",
  "source_type": "local_doc",
  "page_no": 12,
  "slide_no": null,
  "title": "Lecture 5",
  "expires_at": "2026-06-01T00:00:00Z"
}

Tips

  • 不要只把向量存在 Qdrant，Postgres 也要保存 chunk metadata 和 citation mapping。
  • 删除项目时要清 DB、R2、Qdrant，不能只删一边。
  • 给 project_id、document_id、user_id 建索引。
  • 给 expires_at 建索引，方便 7 天自动清理。
  • 不要在 messages 里塞所有运行信息，trace 要独立表。

验收标准

  • [ ] 数据模型能支持 Ask、Quiz、Sources、Trace、Deletion。
  • [ ] 所有用户数据可按 user_id 隔离。
  • [ ] 删除项目可以找到所有相关 DB/R2/Qdrant 数据。
  • [ ] Qdrant payload 足以做 metadata filter。
  • [ ] 有 migration 文件。
  • [ ] 有基础 seed/mock 数据。
  • [ ] 有 7 天过期清理字段和 job 设计。

9. 阶段 7：文档解析与向量化管线

目标

将上传文件转成可检索的 chunks，并写入 Qdrant。

流程

document uploaded
→ create parse job
→ worker downloads from R2
→ parse document
→ extract page/slide/section
→ chunking
→ embedding
→ upsert Qdrant
→ update document status indexed

Tips

  • PDF 用 Docling，DOCX/PPTX 用 MarkItDown。
  • 解析失败要记录 error message，不要只显示“失败”。
  • chunk 要保留 page/slide/section metadata。
  • embedding 要批处理，避免一次请求过大。
  • upsert Qdrant 前先写 Postgres chunk metadata。
  • 每个 job 要可重试。

验收标准

  • [ ] PDF/DOCX/PPTX 至少各有一个样例解析成功。
  • [ ] document status 能从 uploaded → parsing → indexing → indexed。
  • [ ] chunk 写入 Postgres。
  • [ ] vector 写入 Qdrant。
  • [ ] page/slide/section 可追溯。
  • [ ] 解析失败时用户能看到可理解错误。
  • [ ] worker 重启后 job 不丢失或可恢复。

10. 阶段 8：Local Ask 纵向闭环

目标

先跑通最核心功能：本地资料问答。

流程

user asks
→ resolve mode local_only
→ retrieve Qdrant chunks
→ generate grounded answer
→ validate citations
→ save ask_run
→ return Mode + Sources

Tips

  • 先不要做 Web，不要做 Hybrid。
  • 如果没有 indexed documents，必须拒绝回答。
  • 如果 retrieval 结果低置信度，回答要提示“资料中未找到足够依据”。
  • 回答生成必须使用 structured output。
  • citation 必须来自真实 retrieved chunks。

验收标准

  • [ ] 用户可基于上传文件提问。
  • [ ] 回答顶部显示 Mode: Local 和 Sources。
  • [ ] Sources drawer 能打开对应文件片段。
  • [ ] 检索为空时不编造答案。
  • [ ] citation validator 通过。
  • [ ] ask_run、tool_calls、citations 被记录。

11. 阶段 9：Source Mode 与 Hybrid Local-first

目标

实现 Auto / Local / Web / Hybrid 的策略控制。

Source Policy 优先级

slash command
→ UI selected mode
→ strong prompt signal
→ auto router
→ default no-web

Hybrid v1

先查 local
local evidence 足够则不联网
用户明确要求最新/联网/补充资料时才 web

Tips

  • Auto v1 先用规则，不要一开始做复杂 LLM router。
  • Local Only 禁止调用 Web。
  • Hybrid 必须区分 local evidence 和 web evidence。
  • Web API 调用前先检查 quota。
  • Mode 决策要记录在 trace 中，但普通用户只看到 Mode + Sources。

验收标准

  • [ ] /study 强制 local_only。
  • [ ] /search 强制 web_only。
  • [ ] /hybrid 强制 hybrid。
  • [ ] UI Mode 选择能覆盖 Auto。
  • [ ] Auto 默认不联网。
  • [ ] Hybrid local-first 生效。
  • [ ] source policy validator 能防止越权调用工具。

12. 阶段 10：Web Search / Crawl

目标

实现可控、可引用、可缓存的联网搜索。

流程

/search query
→ Serper search
→ select top urls
→ Jina Reader extract
→ Firecrawl fallback
→ cache content
→ web retrieval
→ generate answer
→ cite web sources

Tips

  • Serper + Tavily 做 provider abstraction。

22. 关键反模式

避免这些坑：

  1. 先写 UI，后想 API：最后前后端字段对不上。
  2. 先做 Report：输出复杂，MVP 会拖慢。
  3. 所有任务都用 Agent：Local Ask 和 Quiz 更适合 workflow。
  4. Auto 默认联网：成本、延迟、噪声都会上升。
  5. 只存向量不存 chunk metadata：Sources 无法追溯。
  6. 只做功能不做删除：正式上线处理用户文件很危险。
  7. Trace 给普通用户看：增加认知负担，还可能泄露内部信息。
  8. API key 放前端：严重安全问题。
  9. 没有 eval 就改 prompt：无法知道质量是变好还是变坏。
  10. 等最后才做 CI/CD：集成风险集中爆炸。

## 0. 总原则

ResearchMate 不是一次性 Demo，而是一个需要正式上线的 AI 应用项目。开发流程应遵循：

1. **先契约，后实现**：先定 API、数据模型、页面行为，再写前后端代码。
2. **先纵向闭环，后横向扩展**：先跑通“上传文件 → 解析 → 向量化 → 提问 → 引用回答”这一条链，再扩展 Web、Hybrid、Quiz。
3. **先可靠，后酷炫**：Agent、RAG、MCP、Trace 都要服务于用户任务，不为了显摆而堆技术。
4. **默认安全**：用户文件、API Key、OAuth、对象存储、向量库、Trace 全部按正式上线标准设计。
5. **每阶段有验收标准**：没有验收标准的任务不进入开发。
6. **每周必须有可演示结果**：避免长期只写底层代码，最后无法展示。
7. **成本可控**：默认 Local-first，Web-on-demand，所有外部 API 做限流、缓存和降级。

| Upload | `POST /documents/upload-url` |
| Document | `POST /documents`, `GET /projects/{id}/documents`, `GET /documents/{id}`, `DELETE /documents/{id}` |
| Job | `GET /jobs/{id}` |
| Ask | `POST /ask` |
| Quiz | `POST /quiz` |
| Sources | `GET /runs/{id}/sources` |
| Trace | `GET /dev/traces/{trace_id}` |
| Health | `GET /healthz` |

### API 设计原则

1. 所有业务 API 必须带认证。
2. 所有 project/document/run 查询必须校验 `user_id`。
3. 长任务只返回 `job_id`，不在请求里同步等完。
4. Ask 响应必须包含 `mode` 和 `sources`。
5. Quiz 响应必须包含题目、答案、解析、引用来源。
6. 错误响应必须统一格式。
7. API 版本从 `/api/v1` 开始。

### 统一错误响应

```json
{
  "error": {
    "code": "DOCUMENT_NOT_INDEXED",
    "message": "Document is still being processed.",
    "request_id": "req_..."
  }
}
```

### Tips

- API Spec 不要只写 endpoint，要写状态码、错误码、权限和示例。
- 文件上传不要让前端拿 R2 secret，要让后端生成 signed upload URL。
- Ask 不要直接返回一堆模型文本，要返回结构化字段。
- Developer Trace API 必须只允许开发者或管理员访问。
- OpenAPI 生成后，可以用它生成前端类型或 API client。

### 验收标准

- [ ] 所有 MVP API 都在 `openapi.yaml` 里定义。
- [ ] 每个 API 有 request、response、error examples。
- [ ] 所有需要认证的 API 都标记 auth requirement。

  "project_id": "...",

## 22. 关键反模式

避免这些坑：

1. **先写 UI，后想 API**：最后前后端字段对不上。
2. **先做 Report**：输出复杂，MVP 会拖慢。
3. **所有任务都用 Agent**：Local Ask 和 Quiz 更适合 workflow。
4. **Auto 默认联网**：成本、延迟、噪声都会上升。
5. **只存向量不存 chunk metadata**：Sources 无法追溯。
6. **只做功能不做删除**：正式上线处理用户文件很危险。
7. **Trace 给普通用户看**：增加认知负担，还可能泄露内部信息。
8. **API key 放前端**：严重安全问题。
9. **没有 eval 就改 prompt**：无法知道质量是变好还是变坏。
10. **等最后才做 CI/CD**：集成风险集中爆炸。

Risks

风险与假设

  • Next 使用 canary 版本是为了规避当前 stable 依赖的 PostCSS 漏洞；后续 stable 发布修复后应切回稳定版。
  • 本地 deterministic answer/quiz 是可测试 fallback，不代表真实 LLM 质量。
  • in-memory store 只适合本地测试，不适合生产。
  • 真实 search/crawl provider 需要 source policy、失败隔离、成本限制和引用校验。

Decision gates

停止条件

  • 任何真实 secret 要进入仓库时，必须停止，改用 GitHub Secrets 或部署平台 Secret Manager。
  • Auto 默认要改成联网时，必须先做隐私、成本和安全评审。
  • 普通用户需要访问 Developer Trace 时，必须先重做权限设计与测试。
  • 修改 owner isolation、RLS、signed URL 或删除逻辑时，必须先补隔离和清理测试。
  • CI 任一质量门失败时，不允许部署到生产。

Next Step

如何用 GitHub Actions 提交并部署上线

提交与 CI

  1. 本地创建分支并提交：git checkout -b feature/your-change，修改后运行 npm run check:all。
  2. 推送分支：git push -u origin feature/your-change。
  3. 在 GitHub 打开 Pull Request，进入 Actions 标签查看 quality-gate。
  4. 只有 quality-gate 通过后再合并到 main。

部署方式

当前仓库已有 CI 质量门，还没有绑定真实生产平台。建议先用托管平台接入 GitHub，再把部署密钥放到 GitHub Secrets 或平台 Secret Manager。

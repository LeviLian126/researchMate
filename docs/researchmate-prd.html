ResearchMate 产品需求文档 PRD

版本：v1.0 状态：MVP 需求定稿 目标周期：8-10 周 文档类型：产品需求文档

---

1. 产品概述

1.1 产品名称

ResearchMate

1.2 一句话定位

ResearchMate 是一个支持本地资料与联网资料融合的 AI 研究学习工作台，帮助学生、自学者和知识工作者基于可信来源生成可溯源回答与测验。

1.3 产品目标

ResearchMate 的目标不是做一个普通 ChatPDF，也不是做一个泛化 AI Search，而是做一个可上线、可演示、可解释的 Agent + RAG 应用：

  • 支持用户上传 PDF / DOCX / PPTX。
  • 支持本地资料问答，并展示来源。
  • 支持联网搜索，但默认不开联网以节省 API 成本。
  • 支持 Hybrid 模式，将本地资料与网络资料综合。
  • 支持 /quiz 生成选择题与简答题。
  • 支持管理员/开发者在受控后台查看 Agent trace、工具调用、检索与校验结果，普通用户界面不暴露 Trace。
  • 支持正式登录和用户数据隔离。

1.4 项目目的

ResearchMate 同时服务两个目标：

  1. 真实产品目标：为学生、自学者和知识工作者提供一个轻量、可信、可溯源的 AI 学习/研究工具。
  2. 校招展示目标：展示 Agent 开发、RAG 工程、云端部署、API 安全、异步任务、可观测性、输出校验与评测能力。

---

2. 问题定义

2.1 学生与自学者的痛点

学生在期末复习、论文阅读、自学课程时常见问题：

  • 课程 PDF、PPT、Word 笔记分散，难以快速定位知识点。
  • 大模型能解释概念，但无法稳定基于上传资料回答。
  • 普通 ChatPDF 只能问答，缺少测验、题目、来源追踪。
  • 复习时需要知道答案来自哪一页、哪一张 slide，而不是只得到模型总结。
  • 有时本地材料不够，需要联网补充案例或最新资料。

2.2 知识工作者的痛点

知识工作者做研究、竞品分析、资料学习时常见问题：

  • 需要同时阅读本地材料和互联网信息。
  • 联网 AI 搜索容易引用不透明，难以判断来源质量。
  • 普通搜索需要手动打开多个网页、整理信息、交叉验证。
  • 研究结果需要可溯源，尤其是区分本地资料和网络资料。

2.3 现有工具不足

工具类型问题
普通 ChatGPT可以上传文件和联网，但项目不可控，不适合作为可展示工程系统
ChatPDF 类工具通常只做单一文档问答，缺少联网和 Agent trace
NotebookLM 类工具产品成熟，但不是自研项目，难以展示后端工程细节
Perplexity 类工具联网强，但本地资料融合和学习测验不是核心
自制 RAG Demo很多只做上传文档问答，缺少产品闭环、校验和部署

2.4 ResearchMate 的切入点

ResearchMate 选择一个明确而不过度膨胀的切入点：

一个统一 Ask 入口，通过 Source Mode 控制资料来源，通过 slash command 控制任务类型，并对所有回答和测验提供可溯源证据。

3. 目标用户

3.1 核心用户

大学生 / 自学者

使用场景：

  • 上传课件 PDF / PPT。
  • 让 AI 解释某个知识点。
  • 生成选择题和简答题。
  • 根据上传资料复习 final。
  • 在本地资料不足时联网补充例子。
普通知识工作者

使用场景：

  • 上传内部资料或研究文档。
  • 联网检索外部信息。
  • 对比本地资料与网络资料。
  • 得到带来源的综合回答。
校招面试官 / 技术评审者

关注点：

  • Agent routing 是否清晰。
  • RAG 是否可溯源。
  • 工具调用是否有边界。
  • 是否处理 API key 安全。
  • 是否有异步任务、限流和错误路径。
  • 是否可正式上线，而不是只在本地跑通。

3.2 非目标用户

MVP 不服务以下用户：

  • 企业级多人知识库管理者。
  • 需要复杂权限管理的团队。
  • 需要在线考试系统的老师。
  • 需要自动批改和错题本的完整学习平台用户。
  • 需要浏览器自动操作的 Agent 用户。

4. 产品原则

4.1 Local-first, Web-on-demand

默认优先使用本地资料和已有上下文，不主动联网。只有以下情况才联网：

  • 用户选择 Web mode。
  • 用户选择 Hybrid mode。
  • 用户输入 /search 或 /hybrid。
  • 用户明确要求“联网”“最新”“搜索”“查官网”“查新闻”等。
  • Hybrid 模式下本地证据不足，且用户允许联网。

4.2 Mode 决定资料来源

Mode 只回答一个问题：从哪里取证据？

支持：

  • Auto
  • Local
  • Web
  • Hybrid

4.3 Slash command 决定任务类型

Slash command 回答另一个问题：要做什么任务？

MVP 支持：

  • /study
  • /search
  • /hybrid
  • /quiz

4.4 回答必须可溯源

每次回答顶部只展示：

Mode: Hybrid
Sources: 5 local chunks · 3 web pages

详细来源放在 Sources panel 中展开。

4.5 普通用户界面简单，管理员视图可追踪

普通用户只看到 Mode、Sources、答案和引用，不看到 Developer Trace、tool raw payload、prompt、token/cost 明细或内部错误栈。

管理员/开发者只能通过受控 Admin 页面查看 Trace，且必须经过管理员权限校验：

  • routing decision
  • tool calls
  • retrieved chunks
  • validation result
  • latency
  • token/cost estimate

Trace 内容必须做脱敏处理，不展示 API key、OAuth token、R2 signed secret、完整系统 prompt 或可能泄露用户隐私的原始敏感字段。

Trace 视图不得出现在普通用户侧边栏、普通页面导航或公开路由入口中。

4.6 默认节省 API fee

系统默认不联网，避免每次请求都消耗 Search/Crawl API。Hybrid 采用 local-first 策略。

5. 核心产品形态

5.1 主要页面

普通用户 MVP 页面：

页面说明
Ask统一问答入口，采用成熟 AI 对话产品的双栏布局
Library文件上传、解析状态、文件列表
Quiz查看 /quiz 生成的测验

管理员/开发者页面：

页面说明可见性
Admin Trace Viewer查看检索、工具调用、校验结果和错误信息仅管理员/开发者

说明：Developer Trace 不属于普通用户主要页面，不能出现在普通用户导航、侧边栏或公开入口中。MVP 只做最小 Admin Trace Viewer，不做完整 Admin Console。

暂不做：

  • Billing
  • Team workspace
  • Full settings
  • 完整 Admin console
  • Flashcards
  • Study plan
  • Report editor

5.2 Ask 页面

Ask 页面参考 ChatGPT、Claude、Gemini 等成熟对话产品的信息架构，采用左侧 session 切换 + 右侧对话主区域的布局。

页面结构：

区域说明
左侧边栏Project / session / conversation 切换；可新建、重命名、删除会话
右侧主区域当前会话消息列表、回答区、Sources 展开面板
底部输入区Ask input、slash command、Mode selector、上传快捷入口

Ask 页面包含：

  • 左侧 session 切换侧边栏
  • 右侧 chat message list
  • 输入框
  • Mode selector
  • Slash command 支持
  • 当前 project/document context
  • 回答区
  • Sources panel

右下角 Mode selector 只包含：

Auto / Local / Web / Hybrid

不包含：

  • Quiz
  • Report
  • Summary
  • Flashcards

这些属于任务类型，不属于资料来源。

6. Source Mode

6.1 Mode 列表

Mode说明默认是否联网
Auto系统判断资料来源，默认不联网否
Local只使用上传资料否
Web只使用联网搜索结果是
Hybrid本地资料优先，必要时联网补充条件触发

6.2 Mode 优先级

Source Mode 解析优先级：

  1. Slash command
  2. 用户手动选择的 Mode
  3. 用户自然语言中的强信号
  4. Auto Router
  5. 默认策略：不联网

6.3 Auto Router 规则

Local 信号
  • “根据我上传的文件”
  • “根据课件”
  • “从这份 PDF 里”
  • “复习 final”
  • “只看本地资料”
  • /study
Web 信号
  • “联网查”
  • “搜索一下”
  • “最新”
  • “现在”
  • “今年/今天/本周”
  • “官网/新闻/价格/融资/榜单”
  • /search
Hybrid 信号
  • “结合我上传的资料和网上资料”
  • “根据论文和最新进展”
  • “课件不够的话联网补充”
  • /hybrid

6.4 Hybrid local-first 策略

Hybrid 第一版采用 local-first：

  1. 先检索本地资料。
  2. 如果本地证据足够，则不联网。
  3. 如果用户明确要求最新资料，则联网补充。
  4. 如果本地证据不足但用户未授权联网，提示是否开启 Web。
  5. Web evidence 作为补充，不覆盖本地课程资料。

7. Slash Commands

7.1 /study

作用：强制使用 Local mode，并采用学习问答风格。

示例：

/study 根据这份 PPT 解释 SVM 的核心思想

行为：

  • 只查询本地资料。
  • 不联网。
  • 回答必须引用本地 page/slide/chunk。
  • 如果资料中没有答案，明确说明未找到。

7.2 /search

作用：强制使用 Web mode。

示例：

/search 查一下 NotebookLM 最新功能

行为：

  • 调用 Search provider。
  • 抓取/读取候选网页。
  • 生成带 Web citation 的回答。
  • 不默认使用本地资料。

7.3 /hybrid

作用：强制使用 Hybrid mode。

示例：

/hybrid 根据我上传的论文，结合最新资料解释 RAG 的发展方向

行为：

  • 本地资料优先。
  • 需要时调用 Web。
  • 回答区分 local evidence 与 web evidence。

7.4 /quiz

作用：基于当前项目资料生成测验。

示例：

/quiz 根据这份课件生成 5 道选择题和 3 道简答题

行为：

  • 默认 Local only。
  • 支持选择题和简答题。
  • 每题必须有答案和解析；不要求逐题绑定 source citation。
  • 不做在线答题、自动批改、错题本。

---

8. Quiz 功能需求

8.1 MVP 支持题型

题型支持说明
选择题是4 个选项，1 个正确答案
简答题是标准答案 + 解析
判断题否Phase 2 可加
自适应题库否Phase 2 可加

8.2 Quiz 输出字段

每道题包含：

  • question
  • type
  • options，选择题必填
  • answer
  • explanation
  • difficulty

Quiz 输出不要求在每道题字段内包含 source citations。可在 QuizSet 层保留整体 sources summary，供用户知道本次测验基于哪些本地资料生成。

8.3 Quiz 验收标准

  • /quiz 必须生成结构化题目。
  • 选择题必须有 4 个选项。
  • 每题必须有标准答案。
  • 每题必须有解析。
  • 不要求每题绑定 source citation。
  • 如果项目没有本地资料，应提示用户先上传文件。
  • Quiz 不出现在右下角 Mode selector 中。

Trace 内容必须做脱敏处理，不展示 API key、OAuth token、R2 signed secret、完整系统 prompt 或可能泄露用户隐私的原始敏感字段。

10. Library 功能需求

10.1 支持文件类型

MVP 支持：

  • PDF
  • DOCX
  • PPTX

10.2 上传限制

MVP 默认：

  • 单文件 25MB
  • 限制可配置
  • 文件类型白名单
  • MIME/type 校验

10.3 文件状态

文件状态包括：

  • uploaded
  • parsing
  • parsed
  • indexing
  • ready
  • failed
  • expired
  • deleted

10.4 文件删除

支持：

  • 用户手动删除项目。
  • 用户手动删除文件。
  • 默认 7 天自动过期删除。

删除必须清理：

  • Supabase Postgres metadata
  • R2 objects
  • Qdrant vectors
  • Redis cache/job status

11. 登录与用户隔离

11.1 登录方式

MVP 使用 OAuth 登录：

  • Google
  • GitHub

11.2 用户隔离

每个用户只能访问自己的：

  • projects
  • documents
  • chunks metadata
  • messages
  • quiz sets
  • traces

所有后端接口必须校验 user_id。

11.3 安全原则

  • 文件默认私有。
  • 前端不能直接访问 R2 secret。
  • 用户只能通过 signed URL 上传。
  • 访问文件预览必须经过后端权限校验或短时效 signed URL。
  • Supabase service role key 只能放后端。
  • 所有数据库查询使用参数化查询或 ORM 安全接口，禁止字符串拼接 SQL。
  • 所有登录、OAuth callback、API 请求必须校验 token、issuer、audience、过期时间和用户状态。
  • 所有 project/document/run/quiz/trace 请求必须做 resource owner 校验。
  • 所有用户输入、文件内容、网页内容都视为 untrusted input。
  • Retrieved content 不能覆盖 system prompt、tool policy、source policy 或权限策略。

---

12. 安全校验与防护要求

12.1 输入与 API 校验

所有后端 API 必须执行统一校验：

  • 认证校验：拒绝无效、过期、issuer/audience 不匹配的 token。
  • 授权校验：所有资源访问都必须验证 request.user_id == resource.owner_user_id。
  • Schema 校验：请求体、query 参数、path 参数必须通过 Pydantic/Zod schema。
  • 类型与范围校验：限制字符串长度、数组长度、文件大小、分页大小、mode 枚举值。
  • 速率限制：Ask、Quiz、Upload、Search、Crawl、Trace API 都需要 per-user / per-IP rate limit。
  • 错误脱敏：错误响应不返回 SQL、stack trace、secret、provider raw error。

12.2 SQL 注入防护

数据库访问必须遵守：

  • 禁止使用字符串拼接构造 SQL。
  • 使用参数化查询、prepared statement 或 ORM 安全 query builder。
  • 动态排序字段、filter 字段必须用 allowlist。
  • 所有多租户查询必须包含 user_id / project_id 条件。
  • 管理员接口也必须经过权限校验和审计日志。

12.3 Prompt Injection 防护

ResearchMate 必须默认把以下内容视为不可信：

  • 用户消息。
  • 上传文件内容。
  • PDF / DOCX / PPTX 中隐藏文本。
  • 网页正文、网页标题、搜索结果 snippet。
  • 历史会话内容。

防护要求：

  • Retrieved content 只能作为 evidence，不得作为 instruction。
  • Tool calls 只能由后端 graph / policy 节点决定，不能由网页或文档文本直接触发。
  • Local mode 不得调用 Web tools，即使文档或用户内容要求联网。
  • Web content 不得修改 source policy、tool policy、system prompt、权限规则或输出 schema。
  • 生成前后都要经过 SourcePolicyValidator、Schema Validator、Citation Validator。
  • 检测到提示词注入或越权指令时，应忽略该内容并记录安全事件。

12.4 登录、Session 与 OAuth 防护

  • OAuth callback 必须校验 state / nonce。
  • Session token 必须校验 issuer、audience、signature、expiry。
  • Cookie 模式需要 CSRF 防护；Bearer token 模式需要严格 CORS 和 token 存储策略。
  • 登录失败、异常 token、越权访问要记录审计日志。
  • 普通用户不能访问 Admin Trace Viewer 或 /api/dev/*。

12.5 文件与网页内容安全

  • 上传文件必须做扩展名、MIME、大小三重校验。
  • R2 object key 必须随机化，不能包含邮箱、文件原名中的敏感信息或可预测路径。
  • 不执行用户上传文件中的脚本、宏或嵌入对象。
  • Web reader/crawler 必须限制协议、重定向、内网地址访问，防止 SSRF。
  • 外部网页抓取失败不得进入 citations 或 sources。

12.6 安全验收标准

  • [ ] 普通用户无法访问 Admin Trace Viewer 和 /api/dev/*。
  • [ ] 用户 A 无法访问用户 B 的 project/document/run/quiz/trace。
  • [ ] Local mode 越权联网测试失败即阻断。
  • [ ] SQL 注入 payload 不会改变查询语义。
  • [ ] Prompt injection 样例不会改变 tool policy / source policy。
  • [ ] OAuth callback 缺少或伪造 state/nonce 时被拒绝。
  • [ ] 前端 bundle 不包含 provider secret。
  • [ ] 错误响应不泄露 stack trace、SQL、secret。

  • 报告生成器。
  • 错题本。
  • 自动批改。
  • 自适应学习计划。
  • Flashcards。
  • 浏览器自动操作 Agent。
  • 多 Agent 群聊。
  • 企业多租户权限。
  • 计费系统。
  • 完美 OCR。
  • 移动端 App。

## 1. 产品概述

### 1.1 产品名称

**ResearchMate**

### 1.2 一句话定位

ResearchMate 是一个支持本地资料与联网资料融合的 AI 研究学习工作台，帮助学生、自学者和知识工作者基于可信来源生成可溯源回答与测验。

### 1.3 产品目标

ResearchMate 的目标不是做一个普通 ChatPDF，也不是做一个泛化 AI Search，而是做一个可上线、可演示、可解释的 Agent + RAG 应用：

- 支持用户上传 PDF / DOCX / PPTX。
- 支持本地资料问答，并展示来源。
- 支持联网搜索，但默认不开联网以节省 API 成本。
- 支持 Hybrid 模式，将本地资料与网络资料综合。
- 支持 `/quiz` 生成选择题与简答题。
- 支持管理员/开发者在受控后台查看 Agent trace、工具调用、检索与校验结果，普通用户界面不暴露 Trace。
- 支持正式登录和用户数据隔离。

### 1.4 项目目的

ResearchMate 同时服务两个目标：

1. **真实产品目标**：为学生、自学者和知识工作者提供一个轻量、可信、可溯源的 AI 学习/研究工具。
2. **校招展示目标**：展示 Agent 开发、RAG 工程、云端部署、API 安全、异步任务、可观测性、输出校验与评测能力。

## 2. 问题定义

### 2.1 学生与自学者的痛点

学生在期末复习、论文阅读、自学课程时常见问题：

- 课程 PDF、PPT、Word 笔记分散，难以快速定位知识点。
- 大模型能解释概念，但无法稳定基于上传资料回答。
- 普通 ChatPDF 只能问答，缺少测验、题目、来源追踪。
- 复习时需要知道答案来自哪一页、哪一张 slide，而不是只得到模型总结。
- 有时本地材料不够，需要联网补充案例或最新资料。

### 2.2 知识工作者的痛点

知识工作者做研究、竞品分析、资料学习时常见问题：

- 需要同时阅读本地材料和互联网信息。
- 联网 AI 搜索容易引用不透明，难以判断来源质量。
- 普通搜索需要手动打开多个网页、整理信息、交叉验证。
- 研究结果需要可溯源，尤其是区分本地资料和网络资料。

### 2.3 现有工具不足

| 工具类型 | 问题 |
|---|---|
| 普通 ChatGPT | 可以上传文件和联网，但项目不可控，不适合作为可展示工程系统 |
| ChatPDF 类工具 | 通常只做单一文档问答，缺少联网和 Agent trace |
| NotebookLM 类工具 | 产品成熟，但不是自研项目，难以展示后端工程细节 |
| Perplexity 类工具 | 联网强，但本地资料融合和学习测验不是核心 |
| 自制 RAG Demo | 很多只做上传文档问答，缺少产品闭环、校验和部署 |

### 2.4 ResearchMate 的切入点

ResearchMate 选择一个明确而不过度膨胀的切入点：

> 一个统一 Ask 入口，通过 Source Mode 控制资料来源，通过 slash command 控制任务类型，并对所有回答和测验提供可溯源证据。

## 3. 目标用户

### 3.1 核心用户

#### 大学生 / 自学者

使用场景：

- 上传课件 PDF / PPT。
- 让 AI 解释某个知识点。
- 生成选择题和简答题。
- 根据上传资料复习 final。
- 在本地资料不足时联网补充例子。

#### 普通知识工作者

使用场景：

- 上传内部资料或研究文档。
- 联网检索外部信息。
- 对比本地资料与网络资料。
- 得到带来源的综合回答。

#### 校招面试官 / 技术评审者

关注点：

- Agent routing 是否清晰。
- RAG 是否可溯源。
- 工具调用是否有边界。
- 是否处理 API key 安全。
- 是否有异步任务、限流和错误路径。
- 是否可正式上线，而不是只在本地跑通。

### 3.2 非目标用户

## 4. 产品原则

### 4.1 Local-first, Web-on-demand

默认优先使用本地资料和已有上下文，不主动联网。只有以下情况才联网：

- 用户选择 Web mode。
- 用户选择 Hybrid mode。
- 用户输入 `/search` 或 `/hybrid`。
- 用户明确要求“联网”“最新”“搜索”“查官网”“查新闻”等。
- Hybrid 模式下本地证据不足，且用户允许联网。

### 4.2 Mode 决定资料来源

Mode 只回答一个问题：**从哪里取证据？**

支持：

- Auto
- Local
- Web
- Hybrid

### 4.3 Slash command 决定任务类型

Slash command 回答另一个问题：**要做什么任务？**

MVP 支持：

- `/study`
- `/search`
- `/hybrid`
- `/quiz`

### 4.4 回答必须可溯源

每次回答顶部只展示：

```text
Mode: Hybrid
Sources: 5 local chunks · 3 web pages
```

详细来源放在 Sources panel 中展开。

### 4.5 普通用户界面简单，管理员视图可追踪

普通用户只看到 Mode、Sources、答案和引用，不看到 Developer Trace、tool raw payload、prompt、token/cost 明细或内部错误栈。

管理员/开发者只能通过受控 Admin 页面查看 Trace，且必须经过管理员权限校验：

- routing decision
- tool calls
- retrieved chunks
- validation result
- latency
- token/cost estimate

Trace 内容必须做脱敏处理，不展示 API key、OAuth token、R2 signed secret、完整系统 prompt 或可能泄露用户隐私的原始敏感字段。

Trace 视图不得出现在普通用户侧边栏、普通页面导航或公开路由入口中。

### 4.6 默认节省 API fee

系统默认不联网，避免每次请求都消耗 Search/Crawl API。Hybrid 采用 local-first 策略。

不包含：

- Quiz

## 6. Source Mode

### 6.1 Mode 列表

| Mode | 说明 | 默认是否联网 |
|---|---|---:|
| Auto | 系统判断资料来源，默认不联网 | 否 |
| Local | 只使用上传资料 | 否 |
| Web | 只使用联网搜索结果 | 是 |
| Hybrid | 本地资料优先，必要时联网补充 | 条件触发 |

### 6.2 Mode 优先级

Source Mode 解析优先级：

1. Slash command
2. 用户手动选择的 Mode
3. 用户自然语言中的强信号
4. Auto Router
5. 默认策略：不联网

### 6.3 Auto Router 规则

#### Local 信号

- “根据我上传的文件”
- “根据课件”
- “从这份 PDF 里”
- “复习 final”
- “只看本地资料”
- `/study`

#### Web 信号

- “联网查”
- “搜索一下”
- “最新”
- “现在”
- “今年/今天/本周”
- “官网/新闻/价格/融资/榜单”
- `/search`

#### Hybrid 信号

- “结合我上传的资料和网上资料”
- “根据论文和最新进展”
- “课件不够的话联网补充”
- `/hybrid`

### 6.4 Hybrid local-first 策略

Hybrid 第一版采用 local-first：

1. 先检索本地资料。
2. 如果本地证据足够，则不联网。
3. 如果用户明确要求最新资料，则联网补充。
4. 如果本地证据不足但用户未授权联网，提示是否开启 Web。
5. Web evidence 作为补充，不覆盖本地课程资料。

## 7. Slash Commands

### 7.1 `/study`

作用：强制使用 Local mode，并采用学习问答风格。

示例：

```text
/study 根据这份 PPT 解释 SVM 的核心思想
```

行为：

- 只查询本地资料。
- 不联网。
- 回答必须引用本地 page/slide/chunk。
- 如果资料中没有答案，明确说明未找到。

### 7.2 `/search`

作用：强制使用 Web mode。

示例：

```text
/search 查一下 NotebookLM 最新功能
```

行为：

- 调用 Search provider。
- 抓取/读取候选网页。
- 生成带 Web citation 的回答。
- 不默认使用本地资料。

### 7.3 `/hybrid`

作用：强制使用 Hybrid mode。

示例：

```text
/hybrid 根据我上传的论文，结合最新资料解释 RAG 的发展方向
```

行为：

- 本地资料优先。
- 需要时调用 Web。
- 回答区分 local evidence 与 web evidence。

### 7.4 `/quiz`

作用：基于当前项目资料生成测验。

示例：

```text
/quiz 根据这份课件生成 5 道选择题和 3 道简答题
```

行为：

- 默认 Local only。
- 支持选择题和简答题。
- 每题必须有答案和解析；不要求逐题绑定 source citation。
- 不做在线答题、自动批改、错题本。

## 8. Quiz 功能需求

### 8.1 MVP 支持题型

| 题型 | 支持 | 说明 |
|---|---:|---|
| 选择题 | 是 | 4 个选项，1 个正确答案 |
| 简答题 | 是 | 标准答案 + 解析 |
| 判断题 | 否 | Phase 2 可加 |
| 自适应题库 | 否 | Phase 2 可加 |

### 8.2 Quiz 输出字段

每道题包含：

- question
- type
- options，选择题必填
- answer
- explanation
- difficulty

Quiz 输出不要求在每道题字段内包含 source citations。可在 QuizSet 层保留整体 sources summary，供用户知道本次测验基于哪些本地资料生成。

### 8.3 Quiz 验收标准

- `/quiz` 必须生成结构化题目。
- 选择题必须有 4 个选项。
- 每题必须有标准答案。
- 每题必须有解析。
- 不要求每题绑定 source citation。
- 如果项目没有本地资料，应提示用户先上传文件。
- Quiz 不出现在右下角 Mode selector 中。

## 11. 登录与用户隔离

### 11.1 登录方式

MVP 使用 OAuth 登录：

- Google
- GitHub

### 11.2 用户隔离

每个用户只能访问自己的：

- projects
- documents
- chunks metadata
- messages
- quiz sets
- traces

所有后端接口必须校验 user_id。

### 11.3 安全原则

- 文件默认私有。
- 前端不能直接访问 R2 secret。
- 用户只能通过 signed URL 上传。
- 访问文件预览必须经过后端权限校验或短时效 signed URL。
- Supabase service role key 只能放后端。
- 所有数据库查询使用参数化查询或 ORM 安全接口，禁止字符串拼接 SQL。
- 所有登录、OAuth callback、API 请求必须校验 token、issuer、audience、过期时间和用户状态。
- 所有 project/document/run/quiz/trace 请求必须做 resource owner 校验。
- 所有用户输入、文件内容、网页内容都视为 untrusted input。
- Retrieved content 不能覆盖 system prompt、tool policy、source policy 或权限策略。

## 12. 安全校验与防护要求

### 12.1 输入与 API 校验

所有后端 API 必须执行统一校验：

- 认证校验：拒绝无效、过期、issuer/audience 不匹配的 token。
- 授权校验：所有资源访问都必须验证 `request.user_id == resource.owner_user_id`。
- Schema 校验：请求体、query 参数、path 参数必须通过 Pydantic/Zod schema。
- 类型与范围校验：限制字符串长度、数组长度、文件大小、分页大小、mode 枚举值。
- 速率限制：Ask、Quiz、Upload、Search、Crawl、Trace API 都需要 per-user / per-IP rate limit。
- 错误脱敏：错误响应不返回 SQL、stack trace、secret、provider raw error。

### 12.2 SQL 注入防护

数据库访问必须遵守：

- 禁止使用字符串拼接构造 SQL。
- 使用参数化查询、prepared statement 或 ORM 安全 query builder。
- 动态排序字段、filter 字段必须用 allowlist。
- 所有多租户查询必须包含 `user_id` / `project_id` 条件。
- 管理员接口也必须经过权限校验和审计日志。

### 12.3 Prompt Injection 防护

ResearchMate 必须默认把以下内容视为不可信：

- 用户消息。
- 上传文件内容。
- PDF / DOCX / PPTX 中隐藏文本。
- 网页正文、网页标题、搜索结果 snippet。
- 历史会话内容。

防护要求：

- Retrieved content 只能作为 evidence，不得作为 instruction。
- Tool calls 只能由后端 graph / policy 节点决定，不能由网页或文档文本直接触发。
- Local mode 不得调用 Web tools，即使文档或用户内容要求联网。
- Web content 不得修改 source policy、tool policy、system prompt、权限规则或输出 schema。
- 生成前后都要经过 SourcePolicyValidator、Schema Validator、Citation Validator。
- 检测到提示词注入或越权指令时，应忽略该内容并记录安全事件。

### 12.4 登录、Session 与 OAuth 防护

- OAuth callback 必须校验 state / nonce。
- Session token 必须校验 issuer、audience、signature、expiry。
- Cookie 模式需要 CSRF 防护；Bearer token 模式需要严格 CORS 和 token 存储策略。
- 登录失败、异常 token、越权访问要记录审计日志。
- 普通用户不能访问 Admin Trace Viewer 或 `/api/dev/*`。

### 12.5 文件与网页内容安全

- 上传文件必须做扩展名、MIME、大小三重校验。
- R2 object key 必须随机化，不能包含邮箱、文件原名中的敏感信息或可预测路径。
- 不执行用户上传文件中的脚本、宏或嵌入对象。
- Web reader/crawler 必须限制协议、重定向、内网地址访问，防止 SSRF。
- 外部网页抓取失败不得进入 citations 或 sources。

### 12.6 安全验收标准

- [ ] 普通用户无法访问 Admin Trace Viewer 和 `/api/dev/*`。
- [ ] 用户 A 无法访问用户 B 的 project/document/run/quiz/trace。
- [ ] Local mode 越权联网测试失败即阻断。
- [ ] SQL 注入 payload 不会改变查询语义。
- [ ] Prompt injection 样例不会改变 tool policy / source policy。
- [ ] OAuth callback 缺少或伪造 state/nonce 时被拒绝。
- [ ] 前端 bundle 不包含 provider secret。
- [ ] 错误响应不泄露 stack trace、SQL、secret。

## 14. 成功标准

### 14.1 产品成功标准

- 用户能通过 Google/GitHub 登录。
- 用户能上传 PDF/DOCX/PPTX。
- 文件能进入解析和索引流程。
- Local mode 能基于本地资料回答。
- Web mode 能联网检索并给出来源。
- Hybrid mode 能 local-first，必要时联网。
- `/quiz` 能生成选择题和简答题。
- 每个回答都有可展开 Sources；Quiz 可展示 QuizSet 级别 Sources，不要求逐题 source citation。
- 用户可以手动删除项目。
- 系统默认 7 天自动过期删除匿名/临时项目或用户项目数据，具体可配置。

### 14.2 工程成功标准

- 前端不暴露任何 LLM/Search/Embedding/Qdrant/R2 secret。
- 所有用户数据请求都做 user_id 校验。
- 所有 LLM 输出经过 schema 校验。
- 问答引用都能追溯到 chunk/page/slide/url；Quiz 保留整体资料来源 summary，不强制逐题 citation。
- Search/Crawl 有限流和缓存。
- RAG eval 可以运行。
- Admin-only Developer Trace 能展示 routing、retrieval、tool calls、validation，普通用户不可访问。

## 16. 面试 Demo 脚本

### 16.1 Demo 1：本地资料学习

1. 登录。
2. 上传课程 PPT。
3. 等待解析完成。
4. 输入：`/study 根据这份 PPT 解释 RAG`。
5. 展示 Mode 和 Sources。
6. 展开 Sources，定位到 slide/page。

### 16.2 Demo 2：Quiz 生成

1. 输入：`/quiz 生成 5 道选择题和 2 道简答题`。
2. 展示每题答案和解析，并展示 QuizSet 级别资料来源。
3. 展示结构化输出 schema。

### 16.3 Demo 3：Hybrid

1. 上传一篇论文或课件。
2. 输入：`/hybrid 结合这份资料和最新网络资料解释这个方向`。
3. 展示 local-first 策略。
4. 展示 Sources 中 local 和 web 的区别。

### 16.4 Demo 4：Admin-only Developer Trace

1. 使用管理员/开发者账号打开 Admin Trace Viewer。
2. 展示 SourcePolicyResolver 输出。
3. 展示 Qdrant retrieval。
4. 展示 tool calls。
5. 展示 validation result。
6. 说明普通用户无法从 Ask 页面、侧边栏或普通导航进入 Trace。

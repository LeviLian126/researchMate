This is the field-level source of truth for the maintained SQL migrations: 15 initial tables, 16 additive domain tables, and one backend-only operational table. Local/test still defaults to memory. PostgreSQL adapters and worker repositories exist, but no migration has been applied to Supabase.

Contract status and conventions

Primary tenant keys

user_id and project_id scope projects, documents, chunks, conversations, runs, quizzes and jobs.

Deletion model

Cascade for dependent records; selected run references use ON DELETE SET NULL ; projects/documents/conversations also support soft deletion.

Runtime gap

Schema presence does not prove an applied migration, active RLS, durable API writes or production retention.

Additive migration and repository contracts

Workflow, evidence, report, outbox, evaluation, and fault entities

Fifteen additive contracts come from 202607150002_evidence_review_schema.sql; 202607150005_evaluation_budget_and_fault_exercises.sql adds fault_exercises, bringing the domain total to 31 tables (15 initial + 16 additive). Migration 006 adds one backend-only operational runtime_heartbeats table and workflow lease/budget fields; it is not counted as a domain table. FastAPI and worker repositories use these shapes, but no migration has been applied to Supabase and no managed-Postgres integration has run.

Storage ownership: PostgreSQL is the source of truth for business state. R2 stores private source bytes and parser artifacts; Qdrant stores rebuildable dense/sparse projections; Redis stores broker and short-lived coordination state only. No irreplaceable business fact may exist only in R2, Qdrant, or Redis.
Checkpoint boundary: LangGraph Postgres checkpoint tables are SDK-managed infrastructure, not domain tables. A guarded release command now runs the SDK's versioned setup() once before deployment; workers no longer execute DDL. No checkpoint table exists in a validated managed database, and these four tables are not included in the 31-domain-table count.

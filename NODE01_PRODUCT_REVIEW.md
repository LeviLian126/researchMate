# ResearchMate Node01 产品评审

> 评审日期：2026-07-15
> 评审范围：目标用户、问题、产品承诺、首次成功、MVP 边界、功能价值、验收、市场替代品与下一步验证。
> 证据边界：仓库代码、测试、当前 HTML 文档与公开产品资料；没有真实用户行为、访谈、留存、付费或生产数据。

## 1. 结论摘要

### Node01 结论

**判定：`Promising but unvalidated（方向有潜力，但尚未验证）`。**

ResearchMate 已经证明“项目 → 资料 → 有引用回答 → 来源检查 → Quiz”的本地技术闭环可以实现，但还没有证明以下产品事实：

1. 谁是最迫切的第一类用户。
2. 用户为什么会从 NotebookLM、ChatGPT Projects 或现有笔记/检索流程迁移。
3. “引用 + Trace + Quiz”中的哪一个结果足以形成重复使用。
4. 用户是否愿意为更强的可核验性承担上传、整理和学习新工具的成本。

### 最重要的产品问题

当前产品同时服务两个不同方向：

| 方向 | 主要用户 | 核心结果 | 当前相关功能 |
|---|---|---|---|
| 资料学习助手 | 学生、研究生、专业学习者 | 从私有资料得到可核验解释并完成复习 | Project、Library、Ask、Citation、Quiz |
| 可观测 RAG 工作台 | AI 开发者、评测或运维人员 | 检查路由、检索、工具调用和验证过程 | Source mode、Developer Trace、Execution plan、可靠性候选 |

这两个方向可以共享底层能力，但第一用户、界面、卖点、验收和分发路径不同。继续同时强化两边，会产生“平台先于楔子”的风险。

### 推荐产品主线

**优先选择“基于私有资料的可核验学习助手”作为第一产品楔子。**

暂定用户假设：

> 对需要反复阅读课程材料、论文或专业文档的研究生和专业学习者，ResearchMate 将一组资料转化为可追溯的解释和可作答的复习练习，使用户能快速理解内容并检查每个关键结论的来源。

理由：

- 当前 Ask、引用和 Quiz 已能形成一个面向用户的连续故事。
- Developer Trace 更适合作为内部质量工具，而不是主产品价值。
- “研究者、学习者、分析师、开发者”同时作为目标用户过宽，无法形成一个明确的首次成功。
- 如果创始人真正想做开发者产品，应另行选择该方向，并删除或弱化 Quiz，而不是让两个产品长期共用一个主界面。

### 建议姿态

**`Reduce（收缩）`：先完成一个可信、可观察、可重复的资料学习闭环，再扩展 Web、复杂编排和可靠性平台。**

## 2. 当前产品事实与信心

| 项目 | 当前判断 | 信心 | 证据或缺口 |
|---|---|---:|---|
| 本地技术闭环可实现 | 已确认 | 90% | API 工作流、前端页面、共享契约和 24 项当前测试 |
| 用户可以创建项目并组织资料 | 部分确认 | 75% | 界面与 API 存在，但状态不持久，真实上传不可用 |
| 回答可以绑定本地引用 | 已确认于本地确定性路径 | 85% | Ask、Sources、Citation、Trace 测试 |
| Quiz 可以由资料生成 | 已确认于生成契约 | 80% | 题目、四选项、引用和历史测试 |
| 用户能在首次使用中自然成功 | 未确认 | 25% | 无浏览器可用性观察；界面暴露开发 Token 和 fallback 文本 |
| 用户会重复使用 | 未知 | 5% | 无持久对话、真实留存或第二次使用证据 |
| 存在强烈需求 | 未知 | 10% | 无用户访谈、真实工作流观察、承诺或付费证据 |
| 产品已有清晰差异化 | 未确认 | 20% | 主要能力与 NotebookLM、ChatGPT Projects 高度重叠 |

**综合产品信心：约 35%。** 技术可行性明显高于需求和定位信心。

## 3. 问题与市场替代品评审

### 当前问题假设

仓库表达的问题是：研究资料分散、通用对话缺少稳定项目上下文、AI 结论难以核验、问答与复习工具割裂。

这个问题合理，但目前主要是创始人推断，并非用户行为证据。尚未看到：

- 用户最近一次遇到该问题的真实过程。
- 当前使用哪些工具拼接解决。
- 每周发生多少次、损失多少时间或造成什么错误。
- 哪个时刻会促使用户切换工具。
- 用户是否会因为引用质量而放弃更熟悉的平台。

### 替代品压力

ResearchMate 不能把“上传资料 + 基于资料问答 + 引用 + 学习产物”当作天然差异化：

- [NotebookLM 官方帮助](https://support.google.com/notebooklm/answer/16164461?hl=en)已经提供多类资料上传、基于来源的问答、行内引用、学习指南、简报、音频概览和思维导图。
- [NotebookLM 引用交互](https://support.google.com/notebooklm/answer/16179559?hl=en)可以从引用跳转到原文位置，并允许用户选择参与回答的来源。
- [ChatGPT Projects](https://help.openai.com/en/articles/10169521-using-projects-in-chatgpt-496)已经把文件、对话、项目指令和历史放进持续项目上下文，并支持共享与项目内记忆。

因此当前差异化不能只是：

- “我们也有项目和文件”；
- “我们也能显示引用”；
- “我们也能生成测验”；
- “我们可以接很多模型和搜索提供商”。

### 可测试的差异化假设

优先测试以下一个差异化，而不是同时承诺全部：

1. **引用可核验性更强**：关键结论能快速定位到原文，而不只是显示片段。
2. **学习闭环更完整**：从问题到练习、作答、反馈和薄弱点复习，而不只是生成题目和答案。
3. **来源边界更可控**：用户能明确选择哪些资料参与当前回答，并理解 Web 证据是否被使用。
4. **特定中文专业资料体验更好**：对一个明确学科或证书场景提供更合适的解释和复习路径。

目前仓库没有证据证明其中任何一项成立。

## 4. 当前首次成功评审

### 当前设计的首次成功

创建项目 → 选择文件 → 填写本地解析文本 → 完成处理 → 使用 `/study` 提问 → 查看回答和引用。

### 主要断点

1. **上传承诺与真实行为不一致。**
   - 界面让用户选择 PDF、DOCX 或 PPTX，但当前默认使用预填的 `manualText`，所选文件并未被真实解析。
   - 用户可能误以为回答来自上传文件，实际上来自 fallback 示例文本。
   - 这是当前最高优先级的产品诚信问题。

2. **工作台暴露开发工具。**
   - 项目页要求理解 `dev`、`dev-user-a`、Bearer Token。
   - 这适合开发验证，不适合新手首次成功。

3. **命令和 Mode 是实现概念。**
   - `/study`、`/search`、`/hybrid`、`local_only` 等要求用户先理解路由模型。
   - 普通用户更容易理解“仅使用我的资料”“允许搜索网络”“选择参与的资料”。

4. **引用只展示片段，没有形成核验动作。**
   - 当前来源卡显示类型、页码或 URL 和 quote，但不能打开文档并定位上下文。
   - “有 citation”不等于“用户完成了验证”。

5. **Quiz 是生成结果，不是学习闭环。**
   - 答案和解析立即展示，没有作答、提交、判分、错题或再次复习。
   - 当前更像 Quiz 内容预览，而不是可验证的学习效果。

6. **第二次使用路径不存在。**
   - 项目、资料、回答和 Quiz 在进程重启后丢失。
   - 没有会话历史、恢复、最近活动或继续上次学习入口。

### 推荐的首次成功

> 用户在 10 分钟内导入一份自己的真实资料，提出一个真实问题，获得带可点击原文定位的回答，并确认至少一个关键结论确实得到资料支持。

Quiz 应作为首次成功后的第二价值时刻，而不是首个必要步骤。

## 5. 功能决策

### Keep：保留并强化

| 功能 | 决策理由 | 必须补齐的产品结果 |
|---|---|---|
| Project | 为资料和问题提供稳定边界 | 隐藏开发 Token；明确项目名称、最近活动和继续入口 |
| Library | 是首次成功的必要输入 | 真实解析至少一种格式，或诚实改成“粘贴文本” |
| Grounded Ask | 是主价值交付 | 支持自然提问、清晰回答结构和可核验引用 |
| Citation / Sources | 是信任核心 | 点击引用定位原文上下文；支持选择参与回答的资料 |
| Local-first | 可形成隐私与控制价值 | 改成用户语言，默认仅使用所选资料 |
| 基础失败状态 | 防止错误承诺 | 解析失败、资料为空、无支持证据时给出可恢复动作 |

### Change now：立即优化

| 优先级 | 改动 | 当前问题 | 推荐结果 |
|---:|---|---|---|
| P0 | 修复上传诚信 | 文件选择器并不代表真实解析 | 先真实支持 PDF，或移除文件上传并明确为粘贴文本 Demo |
| P0 | 隐藏开发 Token | 新手必须理解内部身份工具 | 普通体验自动使用演示身份；开发身份切换放到单独开发面板 |
| P0 | 重写来源控制 | `/study` 和 Mode 面向实现 | 使用“仅我的资料 / 我的资料 + Web”并默认安全选项 |
| P0 | 引用定位 | 片段卡片不能完成核验 | 点击 citation 打开来源、突出原文并保留前后文 |
| P0 | 首次成功引导 | 用户需要自行发现 Project、Library、Ask 顺序 | 创建项目后进入三步引导，并显示处理完成后的建议问题 |
| P1 | 持久状态和继续入口 | 第二次使用无法成立 | 保存项目、资料、回答历史；重启或重新登录后可继续 |
| P1 | 用户可见的处理状态 | HTTP 202 实际同步，界面只显示文本状态 | 明确 queued/processing/ready/failed，支持失败原因和重试 |
| P1 | 删除和隐私控制 | API 有删除，主 UI 缺少明确退出路径 | 用户能删除资料/项目并看到删除范围和状态 |
| P1 | Quiz 交互闭环 | 直接显示答案，不产生学习行为 | 作答 → 提交 → 反馈 → 引用核验 → 错题重试 |
| P1 | 真实回答质量评测 | 确定性 builder 只能证明契约 | 建立 20–30 个真实资料问题集，评估支持率、引用正确率和拒答 |

### Defer：验证后再做

| 功能 | 延后理由 | 重新进入条件 |
|---|---|---|
| Web-only / Hybrid 完整提供商 | 会增加成本、安全和定位复杂度；本地资料价值尚未验证 | 用户真实任务持续需要资料外证据，且 local-only 任务已产生重复使用 |
| 多模型或多搜索提供商 | 用户购买的是结果，不是适配器数量 | 单一提供商无法达到已定义的质量、成本或可用性门槛 |
| Conversation graph | 当前没有经过验证的连续对话需求 | 观察到用户必须围绕同一资料连续追问才能完成结果 |
| Node-level reliability UI | 面向开发者，不属于学习者首次价值 | 已选择开发者产品方向，或内部质量团队有明确使用者和决策动作 |
| 多人协作和共享 | 会引入权限、审计和同步复杂度 | 单人工作流出现自然分享需求或付费团队试点 |
| 多格式全覆盖 | 当前未证明每种格式都必要 | 目标用户的真实资料分布要求新增格式 |

### Reject for current MVP：当前 MVP 不做

- 以“支持更多 Provider”作为用户卖点。
- 在没有事件定义和使用者的情况下构建复杂可靠性仪表盘。
- 同时服务学生、研究员、企业分析师和 AI 开发者。
- 在真实资料解析未完成前继续包装成通用文件研究产品。
- 在没有需求证据前设计定价套餐、团队权限或大规模多租户能力。

## 6. 推荐 MVP / Minimum Awesome Product

### 单一产品结果

使用一组私有资料，快速得到可以回到原文核验的解释。

### 必须一起交付的最小可信范围

| 能力 | 范围 | 可失败验收标准 |
|---|---|---|
| 快速进入 | 演示身份或最小登录；不显示开发 Token | 新用户无需解释即可进入创建项目流程 |
| 项目 | 创建、重命名、返回、删除 | 第二次进入仍能找到项目；删除后不可继续访问 |
| 资料输入 | 优先真实 PDF；否则明确粘贴文本 | 系统使用的内容与用户看到的输入一致；解析失败不产生 ready 状态 |
| 处理反馈 | queued/processing/ready/failed + retry | 用户能判断是否可提问，并能从失败状态恢复 |
| Ask | 自然语言问题，默认仅使用选中资料 | 无证据时拒答；有证据时每个核心结论有支持关系 |
| 引用核验 | 引用跳转、原文上下文、来源名称/页码 | 用户可在一次点击后看到支持该结论的原文位置 |
| 持久与继续 | 保存资料、问题和回答历史 | 重启或重新登录后能继续同一项目 |
| 隐私与退出 | 删除资料/项目，说明处理边界 | 用户能发起删除并看到完成或失败状态 |

### MVP 之外

- Web 搜索和复杂 Hybrid。
- 多 Provider 切换。
- 完整 Quiz 学习系统。
- 团队协作。
- 面向用户的 Developer Trace。
- Graph、context health 和 reliability simulation。

Quiz 可以保留为受控实验，但在完成作答/反馈闭环前不要把它称为已经验证的学习能力。

## 7. 建议路线图重排

### Slice 0：验证问题，不扩基础设施

- 找到 5 位符合假设的研究生或专业学习者。
- 使用他们自己的资料观察现有流程。
- 可以人工准备文本或在本地运行，不需要先部署完整云架构。
- 目标：确认他们是否真的在意“可回到原文核验”，以及当前替代方案为什么不够。

### Slice 1：首次成功诚信

- 真实 PDF 或诚实的粘贴文本输入。
- 新手引导与演示身份。
- 自然语言来源控制。
- 可点击引用和原文上下文。
- 明确空状态、失败和恢复。

### Slice 2：重复使用

- 最小认证与持久化。
- 项目、资料和问答历史。
- 对话式追问，仅在用户观察证明必要时实现。
- 删除、保留和隐私说明。

### Slice 3：结果质量

- 真实问题集。
- 引用正确率、结论支持率、拒答正确性和用户核验时间。
- 先用一个检索/模型方案达到门槛，再决定是否需要多 Provider。

### Slice 4：第二价值时刻

- 在 Quiz 与 Web 扩展中只选择一个。
- 如果用户目标是考试和学习，完善 Quiz 作答/反馈。
- 如果真实任务持续需要外部新信息，再完善 Web/Hybrid。

### Slice 5：生产和规模

- JWT、RLS、对象存储、Worker、向量检索和监控。
- 仅在准备让真实用户持续使用时完成对应生产证明。
- 不因目标架构图存在而一次性建设所有基础设施。

## 8. 验证计划

### 最危险的 Must 假设

> 目标用户需要一种比 NotebookLM 或 ChatGPT Projects 更容易核验关键结论的私有资料工作流，并愿意把真实资料和重复任务迁移进来。

### 最便宜的下一步

在 7 天内完成 5 次无指导观察：

1. 用户带来最近实际使用过的一份资料和一个真实问题。
2. 先记录其当前工具和流程，不先演示 ResearchMate。
3. 让用户独立完成创建、输入资料、提问和核验引用。
4. 记录卡点、误解、核验动作和是否需要 Quiz。
5. 会后不提醒，观察 7 天内是否主动提出第二个真实任务或再次使用。

### 成功阈值

- 至少 4/5 用户在 10 分钟内完成首次成功，且无需解释 Token、Mode 或 fallback。
- 至少 4/5 能通过引用正确找到支持结论的原文。
- 至少 3/5 在 7 天内主动带来第二个真实问题、第二份资料或请求继续使用。
- 至少 2/5 愿意接受付费试点、明确的每周使用承诺，或提供等价的组织承诺。

### 失败阈值

- 少于 3/5 能完成首次成功。
- 用户认为 NotebookLM/ChatGPT 已足够，且无法说出一个重要缺口。
- 用户只对演示感兴趣，没有第二次真实任务。
- 引用存在但用户仍无法更快确认答案。

### 结果决策

| 结果 | 决策 |
|---|---|
| 达到首次成功、核验和重复使用阈值 | 进入 Slice 2，建设最小持久化和真实用户运行边界 |
| 首次成功通过但无重复使用 | 收窄到一次性高价值任务，或重新评估问题强度 |
| 核验价值弱但 Quiz 使用强 | 改成明确的资料学习/考试产品，并围绕作答反馈优化 |
| Developer Trace 获得开发者强需求 | 将开发者工作台拆成独立产品方向，不与学习者界面混合 |
| 整体阈值失败 | 停止大规模基础设施扩张，重新选择用户或问题 |

## 9. 决策台账

| 决策 | 状态 | 依据 | 产品影响 | 重新评估条件 |
|---|---|---|---|---|
| 第一楔子暂定为可核验资料学习 | 推荐，待创始人确认 | 当前 Ask + Citation + Quiz 形成最连贯用户路径 | Developer Trace 降为内部能力 | 创始人明确选择开发者产品或用户证据反向支持 |
| 采用 Reduce 姿态 | 推荐 | 需求弱于技术完成度，且替代品强 | 暂停横向能力扩张 | 首次成功与重复使用达到阈值 |
| Local-first 默认保留 | 默认 | 有利于来源控制与隐私叙事 | Web 成为显式扩展而非默认 | 用户任务大多必须依赖新鲜外部信息 |
| Quiz 不作为 MVP 必需 | 推荐延后 | 当前没有作答/反馈和学习效果证据 | 首次成功聚焦 Ask + 核验 | 用户观察显示 Quiz 是重复使用主因 |
| Developer Trace 不作为用户卖点 | 推荐 | 它解决开发质量问题，不解决学习者首要结果 | 从普通导航和首页价值中弱化 | 开发者被正式选为第一用户 |
| 不先建设全部目标基础设施 | 推荐 | 可用人工/本地方式验证 Must 假设 | 节省时间并减少不可逆复杂度 | 准备真实持续用户或验证需要对应能力 |

## 10. 未决产品问题

这些问题不能由 Node02 技术设计替代：

1. 第一用户究竟是学习者还是 AI 开发者？
2. 学习者的具体场景是课程复习、论文阅读、资格考试还是企业培训？
3. 用户最在意的是更快理解、引用可信、复习效果还是资料隐私？
4. 首批 5–10 位用户从哪里触达？
5. 用户当前使用 NotebookLM、ChatGPT、Zotero、Notion 或人工笔记的哪种组合？
6. 什么行为代表用户已经获得价值：确认一个结论、完成一次报告，还是提高一次测验成绩？
7. 用户允许资料被发送到哪些模型或搜索提供商？
8. 项目和资料需要保存多久，删除承诺是什么？

## 11. Node01 交接状态

### 当前状态

**`Ready with named risks`：** 可以继续修复首次成功诚信、简化新手体验和增强引用核验；这些改动不要求先解决商业模式。

**`Blocked`：** 不建议在目标用户和核心结果仍未选择时继续扩张 Web/Hybrid、多 Provider、Graph、可靠性平台、团队协作或完整商业基础设施。

### Node02 可直接接收的范围

- 真实 PDF 或明确粘贴文本的诚实输入边界。
- 引用到来源上下文的定位契约。
- 最小项目/资料/问答持久化与删除语义。
- 用户可见的处理状态、失败和重试。
- 默认仅使用所选资料的权限与来源边界。

### 需要创始人确认后才能成为产品事实

- 第一目标用户。
- 主产品方向：学习助手或开发者工作台。
- Quiz 是否属于第一版核心结果。
- Web/Hybrid 是否是近期用户需求。
- 第一个真实用户渠道和验证时间盒。

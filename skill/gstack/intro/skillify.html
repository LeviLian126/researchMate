定位

这个 skill 负责什么

浏览器类能力给 agent 真实眼睛和手。它们通过持久 Chromium 会话完成点击、截图、登录态测试和跨 agent 协作。

把成功的 scrape 流程转成可复用 browser-skill，未来相同意图可快速运行。 它在 gstack 中的角色不是“万能提示词”，而是一个清晰的工作边界：什么时候触发、先检查什么、产出什么、哪些事情必须交给其他能力处理。

适用场景

  • 某个抓取流程会重复用
  • 想把临时操作固化
  • 需要 script.ts + metadata

工作流程

  • Step 1. 回看成功流程
  • Step 2. 生成脚本
  • Step 3. 写 skill 元数据
  • Step 4. 测试快速路径

输出结果

  • browser-skill
  • 脚本文件
  • 快速抽取入口
边界

避免误用的规则

真实产品文档要讲清楚能力边界。这个页面把 skill 的适用范围和不适用范围放在显眼位置，方便用户判断是否应该调用它。

只固化已验证成功的只读抓取流程。

与其他能力的关系

同类能力包括 browse、devex-review、hackernews-frontpage、open-gstack-browser。实际使用时，先由 landing page 或上游 workflow 判断意图，再把任务交给最贴近目标的 skill。

示例触发

/skillify 帮我处理当前项目中最相关的工作流，并在输出里列出证据、风险和下一步。

阅读建议

先从本页理解 skillify 的职责，再回到索引查看同分类 skill。若要修改这个能力，应优先阅读来源文件 skillify/SKILL.md，确认触发条件、流程和反模式是否仍然一致。
